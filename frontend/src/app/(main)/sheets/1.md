## Sheets Feature – Design Approach

Based on the backend code, the Sheets feature allows users to create curated lists of questions (sheets), join sheets created by others, track their progress (solved + revision completed), and view other members’ progress. Below is the proposed UI/UX design approach. I will wait for your confirmation before producing detailed wireframes.

### 1. Sheets Listing Page (`/sheets`)

**Purpose:** Discover and manage all sheets (public, owned, joined).

**Key data from backend:**
- Paginated list with search, owner filter, sorting.
- Each sheet shows: name, description, owner, participant count, creation date.
- Personal status (if authenticated user has joined: show progress summary or “joined” badge).

**UI/UX approach:**
- **Layout:** Clean grid/card layout. Each card has sheet name (heading), description (truncated), owner avatar + name, participant count, and a “Join” button (if not joined) or “View” button (if joined/owned).  
- **Filter bar:** Search input, sort dropdown (by name, latest, most participants), optional owner filter (if viewing own sheets).  
- **Empty state:** “No sheets found. Create your first sheet →” with a prominent call‑to‑action.  
- **Loading/error states:** Skeleton cards, toast for errors, retry option.  
- **Responsive:** 3 cards per row on desktop, 2 on tablet, 1 on mobile.

**User actions:**
- Click card → go to sheet details page.
- Join button → triggers join sheet modal (requires target date picker – future date). After join, user is redirected to sheet details with their progress initialized.
- Create new sheet button (floating or top‑right) → opens create sheet flow (see below).

---

### 2. Sheet Details Page (`/sheets/[slug]`)

**Purpose:** View sheet content (questions grouped by tags), participant list, and personal progress.

**Key data from backend:**
- Sheet metadata (name, description, owner, dates).
- Questions grouped by tags (tag groups).
- List of participants (avatars, usernames).
- Current user’s progress (solved and revisionCompleted counts, target date, completion status).
- Per‑question participant/solved counts (optional).

**UI/UX approach:**
- **Hero section:** Sheet name, description, owner info, participant avatars (with count), join/leave button (for non‑owner), update target date (for member), edit/delete (for owner).
- **Progress summary card:** For the current user, show:
  - Target date (with days left / overdue indicator).
  - Progress rings/bars for **solved** and **revision completed** separately.
  - Overall completion percentage.
  - A “View full progress” link (goes to `/sheets/[slug]/progress/me`).
- **Questions section:** Grouped by tags. Each question displays:
  - Title (clickable link to question detail page).
  - Difficulty badge, platform icon.
  - For current user: icons indicating solved (✅) and revision completed (🔄) status.
  - For all participants: a small badge showing how many participants have solved this question (e.g., “12/20 solved”).
- **Participant list:** Expandable/collapsible section showing participant avatars, usernames, and (for owner) ability to remove members (optional, not in backend yet). Clicking on a participant takes to their progress page.
- **Actions:**
  - Join button (if not joined) → opens target date modal.
  - Leave button (if joined) → confirmation modal.
  - Update target date (pencil icon) → inline date picker.
  - For owner: Edit sheet (name, description, questions) and Delete sheet (soft delete) with confirmation.

**Edge cases:**
- If user not logged in, show only public info and prompt to login to join.
- If sheet has no questions → show empty state.
- If user hasn’t joined, hide progress and question statuses.

**Responsive:** Stack hero and questions on tablet/mobile; tag groups become accordions.

---

### 3. User Progress Page (within a sheet) – `/sheets/[slug]/progress/[username]`

**Purpose:** Detailed view of a specific user’s progress on a sheet (including chart).

**Key data from backend:**
- User membership details (joined at, target date, completed at).
- Per‑question progress (solved, revisionCompleted, last updated).
- Chart data (solved vs revision progress percentages).

**UI/UX approach:**
- **Header:** Username, avatar, sheet name, target date, completion badge.
- **Progress charts:** Dual donut charts or side‑by‑side bars for solved and revision completion percentages.
- **Question list:** Same grouping by tags, but only showing this user’s status (solved/revisionCompleted). Each question shows status icons and last updated date.
- **Share link:** A “Copy link” button to share this progress page.
- **Back navigation:** Breadcrumb or back button to sheet details.

**Empty/error states:** If user hasn’t joined → 404 style message.

---

### 4. Create / Import Sheet

**Manual creation:** Form with sheet name, description, target date (for owner), and a way to add questions (search by title/slug/ID – reusable component). Questions can be added/removed dynamically. Validation ensures no duplicate sheets per owner.

**Import from file:** Upload Excel/CSV/JSON. Preview parsed questions before confirming creation. Show unresolved identifiers with suggestions. Support for large imports with progress indicator.

**UI approach:** Step‑by‑step wizard or modal with two tabs (Manual / Import). After successful creation, redirect to sheet details.

---

### 5. Additional Features to Surface

- **Update target date** – inline on sheet details (for any member) with a date picker, validation (must be future, not before join date).
- **Leave sheet** – confirmation modal.
- **Edit sheet** (owner only) – inline editing of name/description, plus add/remove questions (with same question resolution logic).
- **Delete sheet** – soft delete, confirmation modal.
- **Notifications** (optional) – when a member completes a sheet, perhaps a toast or notification (backend not yet, but frontend can show a message).

---

### 6. Information Hierarchy & Visual Cues

- **Primary action:** Join sheet (if not joined) → most prominent.  
- **Secondary actions:** Update target date, leave, edit/delete (owner).  
- **Progress emphasis:** Use color‑coded badges and progress rings to draw attention to completion status.  
- **Question grouping:** Tags as section headings with horizontal rule, questions listed as river‑style cards (consistent with existing `GoalQuestionItem` style).  
- **Participant avatars:** Small cluster with +X tooltip, click to view profile progress.

---

### 7. Handling Edge Cases Gracefully

- **No questions in sheet:** Show message “No questions added yet” with action for owner to edit.
- **User not joined:** Hide progress info, show “Join this sheet to track your progress” call‑to‑action.
- **Target date passed:** Show “Overdue” badge, still allow updates.
- **Unresolved identifiers during create/import:** Show list of unresolved items with suggestions (typo corrections). Allow manual override.
- **Loading/empty/error states** for all async operations.

---

### 8. Responsive Behaviour

- **Desktop (≥940px):** Two‑column layout on details page (hero + questions on right? Actually hero full width, then questions in two columns? I'll decide in wireframes).  
- **Tablet (768–939px):** Single column, cards stack, tag groups as accordions.  
- **Mobile (≤767px):** Compact, touch‑friendly buttons, avatar rows wrap.

---

### 9. Alignment with Existing Design System

- Use the same **river‑style question items** as in Goals (`GoalQuestionItem` component).  
- Use existing **progress ring** and **progress bar** components.  
- Use **card styles**, **buttons**, **modals**, **toasts** from shared library.  
- Maintain Wabi‑Sabi aesthetic: soft backgrounds, subtle shadows, calm colors.
